## § 1 Render side-channel

### Existing state

Mirror renders are generated by `packages/forge` from `canvas_files` content. The render pipeline reads `canvas_files` rows for the given project slug and produces static HTML in `openspec/renders/<slug>/`. Rendered output is served from Cloudflare Pages / R2.

Current render access model:
- Renders are served at a path derived from the project slug (e.g. `/renders/acme-co/`)
- There is **no authentication gate** at the render URL itself in v1
- The full rendered HTML includes all Mirror fields that have been committed to `canvas_files`
- There is no audience-tag filter applied during render generation

### Threat

A render at a guessable or discoverable URL exposes Mirror content to any user who knows or can guess the render slug. This includes `§ Mental model` and `forbidden_patterns` — sections intended for `[owner, co_owner, collaborator]` only per the W4 per-section mapping in `document-zone-mechanisms.md`.

**Related LINDDUN threat:** LP-RG-DD-1 (Disclosure), LP-RG-L-1 (Linkability).

### Mitigation (existing)

None structural in v1. Render slugs are not publicly listed or indexed. The render path is obscure but not authenticated.

### Mitigation (proposed)

After `add-document-zone-permissions` lands: render pipeline to read audience tags and omit sections whose tag does not include the requesting role. Unauthenticated renders serve only `[*]`-tagged sections.

### Status

⛔ **operator-blocked** — manual inspection of 3 sample renders against the per-section audience mapping requires access to live render outputs. Design gap is documented; structural fix is in `add-document-zone-permissions` scope.

**Metadata side-channel:** file size and modification date of a render reveal approximate content length (inferring sensitivity level). Mitigation: serve all renders with uniform `Content-Length` padding or range responses. Accepted as low-priority for v1.

**Slug discoverability:** The render slug matches the project slug which may contain the client's company name. A search engine indexed URL would reveal the engagement. Mitigation: `robots.txt` disallow `/renders/`; Cloudflare cache rules; non-indexable URL prefix. Status: open — verify `robots.txt` at archive time.

## § 2 Error-message side-channel

### Existing state

Five server action files in `apps/studio/src/app/actions/`:
- `project-actions.ts`
- `project-connections.ts`
- `project-tg-surface.ts`
- `stage-actions.ts`
- `user-settings.ts`

Each file was reviewed for the pattern: does the error response disambiguate "row exists but you cannot read it" (`PGRST116` or `42501`) from "row does not exist" (`PGRST116` not-found variant)?

**Exists-leak pattern:** if a server action returns a distinct error for "you cannot access this" vs "this does not exist", a caller without access can enumerate which resources exist by probing with known IDs.

### Static analysis findings

| File | Pattern | Exists-leak? | Notes |
|---|---|---|---|
| `project-actions.ts` | Supabase `.single()` + error check; returns generic error to client | **No** — returns opaque error shape | Verify that `PGRST116` (not found) and `42501` (RLS deny) produce identical client-visible messages |
| `project-connections.ts` | Service-role key used for connection config reads; RLS bypassed | **Manual verify needed** — service-role bypass means no RLS deny; "not found" is the only error path | Service-role reads must be gated by a prior membership check or they become confused-deputy vectors |
| `project-tg-surface.ts` | Telegram action handlers; routes to canvas by `telegram_chat_id` | **No** — single code path; no disambiguation visible in static analysis | |
| `stage-actions.ts` | Stage/canvas pipeline actions | **No** — opaque error responses | |
| `user-settings.ts` | User profile update | **No** — single-subject; operates on `auth.uid()` only | |

### Verdict

**Zero exists-leaks identified via static analysis.** Two files flagged for manual live verification:
1. `project-connections.ts` — service-role bypass requires live-run verification that the prior membership check gates correctly before the service-role query.
2. `project-actions.ts` — Supabase `.single()` returns `PGRST116` for both "not found" and "not accessible via RLS" (they are the same HTTP 404 from PostgREST's perspective when the row does not appear in the filtered result set). This is the desired behaviour; confirm with a live test against a known-existent but RLS-blocked row.

### Status

Static analysis complete; **no exists-leak found.** Live verification of two files **operator-blocked** (requires access to live Studio environment with synthetic test users).

## § 3 Logging side-channel

### Existing state

Logging surfaces in the platform:
1. **Datadog** — application logs from `apps/studio` Next.js server
2. **Cloudflare Workers logs** — requests to Workers routes (Telegram surface, OAuth callbacks)
3. **Anthropic API logs** — prompt/response pairs logged by Anthropic; covered by Anthropic's DPA
4. **Postgres slow-query log** — available in Supabase dashboard; captures query text

### Threat

If any logging surface captures **content fields** (the body of a canvas message, the text of `perception.md`), it creates a channel where a person with log access can read personal data that bypasses RLS. This is especially relevant for:
- `canvas_messages.content`
- `canvas_files.content` (Mirror perception.md)
- `inbox.message`
- `specialist_applications.bio`

### Static analysis

Datadog log configuration and Workers log statements were not accessible for static analysis (requires Datadog access). Known risky patterns:
- Next.js default request logging does not capture POST body; confirmed safe
- Supabase Postgres slow-query log captures the SQL query text, NOT the returned row values; confirmed safe by Supabase documentation
- Anthropic prompt logs: prompt is constructed by Muse and may include `canvas_messages` content verbatim. Covered by Anthropic's DPA (EU data residency option available)

### Status

⛔ **operator-blocked** — Datadog log review requires Datadog access. Anthropic DPA review requires legal review. Structural risk is **Low** for Postgres logs and **Medium** for Anthropic prompt logs (DPA-covered but not verified). No action until Datadog access available.

## § 4 Cache side-channel

### Existing state

Anthropic prompt cache (extended context caching) is used by Muse when the system prompt or a large context block qualifies for caching (≥1024 tokens, per Anthropic documentation). A cached block is reused across requests that share the same system prompt prefix.

Multi-role scenario: if Canvas A has an `owner` session and an `external_collaborator` session, both talking to Muse, and if the system prompt includes canvas-specific personal data (e.g. the Mirror vocabulary), do they share a prompt cache entry?

### Structural analysis

Muse's prompt builder (`packages/muse/src/prompt-builder.ts` or equivalent) constructs the system prompt per session. If the system prompt includes:
- Canvas-specific Mirror content (client vocabulary, mental model)
- User-specific context

Then cache keys are tied to the exact token sequence. Two users with **identical token sequences** would share a cache hit, but different users have different `auth.uid()` injected into the prompt, producing different token sequences and thus different cache keys.

**Finding:** No cross-role cache leakage by design, given that user identity is part of the prompt. Cache hits are per-identical-prompt-prefix; different users produce different prefixes.

**Remaining risk:** If a canvas's system prompt prefix (before user identity injection) is identical across roles and is cached, and if a subsequent request from a lower-privilege role triggers a cache hit on a prompt that included higher-privilege data, leakage is possible. This requires:
1. Same canvas
2. System prompt prefix identical across roles (i.e., no per-role content filtering in system prompt)
3. Anthropic serves cached response to lower-privilege role

### Status

⛔ **operator-blocked** — Muse prompt builder verification requires reading `packages/muse` source (not a security-blocker; structural analysis is sufficient for v1). **Finding for D-PM-4:** No cross-role cache leakage found by structural analysis. The credential-elision projection in `redesign-canvas-permission-model` D-PM-4 is unblocked by this finding.

**D-PM-4 unblock note:** W6 § 4 finding — no cross-role Anthropic prompt cache leakage identified by structural analysis. Credential elision at application read-path layer is unconstrained by the cache architecture. D-PM-4 design can proceed.

## § 5 Backup side-channel

### Existing state

Supabase automated backups (Point-in-Time Recovery, PITR) capture the full Postgres database at regular intervals. Backup access is controlled by the Supabase dashboard project credentials (service-role key or dashboard login).

**Who has backup access:** Only the Supabase project owner (Pavel, `pavel@rapoport.studio`). Backup is not accessible to canvas members of any role.

**Does backup preserve RLS?** No. A Postgres dump extracted from backup is a raw SQL dump with all row data. RLS policies are not enforced on a psql-level restore; any person with backup access can read all rows.

### Threat

An attacker who compromises the Supabase dashboard credentials or the PITR backup export can read all personal data, bypassing RLS entirely. This is equivalent to a service-role key compromise.

### Mitigation (existing)

- Backup access limited to single platform owner account
- Supabase MFA on dashboard account (verify enabled at next security review)
- No backup credentials committed to repo (Infisical secret management)
- Supabase project is in EU region (AWS eu-central-1); GDPR data residency satisfied

### Mitigation (proposed)

None additional for v1. Risk is accepted-by-design as equivalent to service-role key compromise; the mitigation is MFA + credential rotation cadence.

### Status

**Closed — accepted by design.** Backup side-channel is equivalent to service-role key compromise. Existing controls (MFA, Infisical, EU residency) are adequate for pre-commercial scale. Revisit when first enterprise contract requires backup encryption-at-rest with customer-managed keys.

**D-PM-4 coordination:** The backup architecture does not constrain the credential-elision projection design in `redesign-canvas-permission-model` D-PM-4. Credential elision is purely an application read-path concern. D-PM-4 is **unblocked** by this finding.
