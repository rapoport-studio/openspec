## Aggregate severity assessment

| Severity | Count | Capabilities |
|---|---|---|
| **High** | 4 | Forge Console UI (#10), Project Form Components (#13), Create Project Page (#14), Analytics Standard (#16) |
| **Medium** | 4 | Auth (#2), Agents (#6), Infra (#7), Edit Project Page (#15) |
| **Low** | 8 | Design (#1), API (#3), Platform (#4), FacadeTrack (#5), Contracts (#8), Worker App (#9), UI Components (#11), Entities (#12) |
| **Total** | 16 | 12 module + 4 standalone |

### What this tells us about Vivod's spec discipline

Vivod's spec corpus is **substantially aligned with reality on the platform's load-bearing surfaces** — Design tokens, the API / query-layer / auth wrappers, the Worker App route shell, the Entities registry, and the UI primitives all map cleanly to code at the anchored commit, with only minor scope or location ambiguities. This is well above the median for a 16-capability corpus authored alongside an actively-shipping product.

Where drift concentrates is **at the proposal / pre-apply frontier**: three of the four HIGH-severity drifts (Project Form, Create Project Page, Analytics Standard) describe a *post-change* state that the code has not yet reached. The fourth HIGH drift (Forge Console UI) is a Phase 3 design doc that the code has implemented in a narrower form, suggesting either deliberate scope-cut or an in-flight rewrite. The pattern is consistent: **Vivod writes specs ahead of code, and the deltas track**. The working tree at the anchor commit literally contains an untracked `simplify-project-creation/` change folder and deletions that are converging the project-form code toward the standalone-spec promise — this is the canonical OpenSpec "propose → apply → archive" lifecycle in active motion, not unmanaged drift.

The MEDIUM-severity findings cluster around two themes worth surfacing at kickoff: (a) **agent / worker classification boundaries** (GUARD has spec status but no CI deploy and no `@vivod/agents` package surface; FORGE has rich code but no agent registry entry); (b) **config-source-of-truth gaps** (api-gateway `wrangler.toml` declares no `routes`, so the studio cannot reproduce DNS state from version control alone). Both are tractable conversations, not symptoms of process breakdown.

The LOW-severity findings are almost entirely documentation polish (route-segment ordering, subsection renames, dual-renderer under-documentation) and should not change the studio's confidence in the spec corpus as a working contract for the engagement.

**Implication for the studio's convention adapter (Phase 6 of `onboard-vivod`):** Vivod's specs are reliable enough that the adapter can treat `openspec/specs/<capability>/spec.md` (or `index.md`) as authoritative source-of-truth for "what should be there." When Forge plans against a Vivod spec, the gap between spec and code is most likely a pending Vivod-side change, not a Vivod-side bug — the adapter should surface this as a question to Vivod ("there is an open change `<slug>` that touches this; should this Forge plan stack on top of it?") rather than silently proposing duplicate work.

## Sharing decision

**Studio-internal by default.** Per Phase 2 T2.3 and the design.md § Drift audit scope, the primary purpose of this audit is to give the studio a defensible position at the kickoff conversation — "we read your specs, we read your code, here are the seams we noticed" — not to grade Vivod's existing discipline.

Sharing with Vivod at kickoff is a strategic call (TBD). The audit is written to be shareable as-is (no internal-only language, no judgment beyond observed facts) so the cost of flipping the decision later is low. The natural moment to share is **after charter signing, before the first Forge run**, alongside the convention-adapter design hand-off — at that point the audit serves as a calibration anchor for how aggressively the studio will rely on Vivod's specs to plan work.

Frontmatter carries `shareability: studio-internal-default` until that call is made.
