Архитектура спека-документа

Как устроен чат, как живут идеи и что происходит со спеком по дороге — от первой идеи на онбординге до записи в current/.

1О документе

Зачем он, что описывает и как его читать.

Ядро воркспейса студии сегодня — это «канвас»: таблица canvas_sessions и маршрут /spec/[id]. Название историческое и не отражает того, чем эта сущность является на самом деле. Документ фиксирует правильную модель — данные, жизненные циклы и сквозной флоу — на которой строится вся дальнейшая работа.

Документ описывает текущее состояние (как работает сейчас) и целевую архитектуру (как должно работать). Это аналитический артефакт — основа для будущего OpenSpec-change на ре-архитектуру; формальный design.md будет дистиллирован из него.

Легенда статусов решений

✓ подтверждено — решено в обсуждении ◆ рекомендация — принимается по умолчанию, можно оспорить ? открыто — требует ответа до начала реализации

2Резюме

Главное за тридцать секунд.

Суть ре-архитектуры

Плоская «канвас»-модель заменяется чёткой иерархией из четырёх уровней:

project → spec → idea → { messages, files }

  • Project — engagement: клиент, репозиторий, команда.
  • Spec — живая спецификация проекта, «что истинно сейчас».
  • Idea — предложенное изменение спека. Одна идея = одно изменение = одна ветка = один PR.
  • Messages / Files — диалог по идее и её артефакты.

Ключевое: idea, change и epic — это одна сущность на разных стадиях, а не три таблицы. Доска стикеров (canvas_ideas) удаляется. Все 12 таблиц canvas_* переименовываются.

Ключевая идея словаря

idea, change и epic — это одна сущность (ideas) в разных стадиях и конфигурациях. «Change» — стадия зрелости. «Epic» — идея, у которой есть дочерние идеи. Никаких параллельных таблиц.

4Проблема текущей модели

Что именно не так с тем, как устроено сейчас.

Сегодня в базе 12 таблиц с префиксом canvas_, с корнем в canvas_sessions. У модели три структурные проблемы.

4.1 Неверные имена

СегодняВ чём ошибкаДолжно быть
canvas_* «Холст» — слово ничего не описывает. idea / spec
..._sessions Не сессия. Сущность живёт неделями, копит стоимость, мержится PR-ом. ideas
mode='project' Перегружает слово — рядом есть таблица projects. mode = draft | branched | …

4.2 Размытая иерархия

Нет уровня «спек» как такового. «Канвас» одновременно притворяется и одним изменением, и проектным воркспейсом. На деле один канвас = одна git-ветка canvas/<slug> = один PR = одна папка openspec/changes/<slug>/ — то есть это уже одно изменение, просто названное неправильно.

4.3 Слово «идея» перегружено

В схеме canvas_ideas — это стикеры на спатиальной доске (x/y-сетка, цвет, без жизненного цикла). В целевой модели «идея» — это единица изменения. Два разных понятия под одним словом; одно из них должно уступить.

6Сущности подробно

Что каждая таблица хранит и за что отвечает.

6.1 Project — engagement

Контейнер верхнего уровня. Существует сегодня как таблица projects и остаётся без структурных изменений. Несёт идентичность engagement-а: название, slug, владелец, статус (active / archived). Привязки к GitHub-репозиторию, Linear-команде и Mirror-слою — опциональные «способности» проекта, а не предусловия.

6.2 Spec — живая спецификация

Корень содержательного дерева. Отвечает на вопрос «что истинно сейчас» и физически соответствует openspec/current/ — набору capability-документов. У проекта ровно один спек. Идеи определяются относительно спека: каждая идея — это дельта против него.

? Открыто — см. раздел 17, вопрос 1

Уровень spec сегодня в БД отсутствует. Решение: завести тонкую таблицу specs (строка-якорь) или оставить спек чисто git-деревом. Рекомендация — тонкая таблица.

6.3 Idea — единица изменения

Центральная сущность. Соответствует сегодняшней canvas_sessions после переименования. Одна идея = одно предложенное изменение спека. Несёт:

ПолеНазначение
id, slug, titleИдентичность идеи.
spec_idFK на спек, против которого идёт изменение.
parent_idСамо-ссылка; заполнена у дочерней идеи эпика.
statusСтадия жизненного цикла — см. раздел 7.
sourceПроисхождение: brief / chat / agent / manual.
stage, stage_percentКонвейерная стадия и процент готовности (из текущей модели).
GitHub-поляgithub_branch, github_repo — появляются, когда идея материализуется в ветку.
cost_cents, cost_cap_centsНакопленная стоимость и кэп — см. раздел 14.

6.4 Message — реплика диалога

Соответствует canvas_messages. Принадлежит идее (FK idea_id). Несёт role (user / assistant / system), content, монотонный sequence внутри идеи, author_id и kind. Сообщение может ссылаться на конкретный файл (поле file_ref) — так в диалоге обсуждаются артефакты, отдельная таблица для этого не нужна.

6.5 File — артефакт идеи

Соответствует canvas_files. Принадлежит идее. Несёт path (например openspec/changes/<slug>/proposal.md), content, status (см. раздел 8) и git-якоря (base_sha, base_content) для diff-а и обнаружения конфликтов в режиме ветки.

7Жизненный цикл идеи

Одна строка ideas с полем status; стадии ложатся на три папки OpenSpec.

captured→ discussing→ drafted→ applied

Ветвление: из drafted идея уходит либо в applied (в спек), либо в epic (порождает дочерние идеи). Из discussing возможен выход в rejected.

СтадияЧто происходитГде живёт
captured Идея появилась на таймлайне спека, ещё не обсуждалась. Сюда попадает бриф с онбординга. строка ideas
discussing Идёт диалог с Muse в чате идеи. ideas + messages
drafted У идеи есть файлы proposal/design/tasks — это уже формальный change, материализованный в openspec/changes/<slug>/. ideas + files + папка changes/
applied Транзакция влила файлы в спек (openspec/current/), идея ушла в archive/. current/ + archive/
epic Идея слишком крупная — вместо применения порождает дочерние идеи. ideas + дочерние строки
rejected Идея не пошла. строка ideas

Перевод стадий на язык разговора

  • «первая идея с онбординга» → captured
  • «обрабатывается в чате» → discussing
  • «у идеи появляются файлы, Muse предлагает» → discussing → drafted
  • «транзакция добавляет файлы в current» → drafted → applied
  • «или в epic» → drafted → epic

8Жизненный цикл файла

У файла свой мини-цикл — Muse не пишет файл молча, он предлагает.

proposed→ accepted→ обсуждается / правится→ applied
СтадияЧто это
proposed Muse сгенерировал файл и предлагает добавить его к идее. Человек видит карточку Approve / Reject.
accepted Человек принял — файл стал частью черновика идеи (сегодняшний аналог: canvas_files.is_pending=true).
applied Транзакция применения идеи вписала файл в спек.
rejected Предложение отклонено.

Паттерн «агент предлагает — человек подтверждает» уже спроектирован в эпике Ideas (agent_proposals, карточка IdeaProposalCard с Approve/Reject) и переезжает в эту модель без изменений.

9Чат как таймлайн спека

Чат — это лента идей спека по времени.

Таймлайн спека
СПЕК  (чат-поверхность)
│
├─ idea #1   бриф с онбординга      captured → discussing → drafted → applied
├─ idea #2   «добавить страницу X»  discussing
├─ idea #3   «переписать модель Y»  epic ──┬─ idea #3.1
│                                          └─ idea #3.2
└─ idea #4   …                      captured
  • На уровне спека «чат» — это упорядоченный по времени список идей (ideas ORDER BY created_at). Первая идея — бриф с онбординга.
  • Открываешь идею → внутри её собственный диалог (messages) и её файлы (files).
  • Новые идеи добавляются на ту же ленту, обсуждаются, дозревают до changes.
Отклонённая альтернатива

Один сплошной поток сообщений на спек, а идеи — как «якоря» на нём. Отклонено: такой подход размывает идею по диапазону сообщений. Сообщения принадлежат идее.

10Роль Muse

Как AI-архитектор участвует в жизни идеи.

Muse — AI-субстрат студии. В контексте спека он работает как Muse Architect: ведёт диалог, генерирует файлы, предлагает идеи. Muse — участник, а не автомат: каждое его действие, меняющее состояние (файл, идея), проходит через подтверждение человеком.

Действие MuseОписание
Онбординг-ход Первый ход при открытии идеи-брифа: Muse переформулирует бриф и задаёт уточняющий вопрос — на языке клиента.
Ход диалога Сообщение пользователя → ответ Muse. Может сопровождаться генерацией файлов и предложением идей.
Генерация файла Muse эмитит файл со статусом proposed; человек принимает или отклоняет.
Предложение идеи Muse может предложить новую идею (source=agent) — она появляется на таймлайне в состоянии ожидания решения человека.
Цитирование спека Ответы про capability-спеки несут цитаты-ссылки в текущий спек. Поисковый путь ограничен ZDR (без персистентности).

Каждый ход и каждый вызов инструмента тарифицируется и списывается на идею (см. раздел 14).

11Модель доступа

Кто что видит и кто что может менять.

Доступ строится по цепочке участников проекта. Роли:

РольПрава
ownerПлатформ-владелец. Полный доступ ко всем спекам и идеям проекта.
collaboratorВнутренний участник. Читает и пишет идеи, ведёт диалог.
external_collaboratorСпециалист, нанятый в engagement. Доступ с датой окончания.
clientКлиент. Ограниченный доступ — видит свой спек и диалог, но не служебные поверхности.

У приглашённых также есть персона (architect / domain / tech / business / ops) — она задаёт ракурс, в котором участник вовлечён.

◆ Рекомендация

Таблицы доступа (canvas_invitees, canvas_excluded_members) переезжают на уровень спека, не идеи: клиента приглашают в проект/спек целиком, а не в отдельную идею. Точная привязка — под-дизайн ре-архитектуры.

12Сквозной флоу

Конкретный сценарий: от онбординга до записи в спек.

Путь идеи
бриф        приём        первая идея     диалог        файлы         транзакция
лида    ─▶  лида     ─▶  в спеке     ─▶  с Muse    ─▶  от Muse   ─▶  применения
(inbox)     (project+    (captured)      (discussing)  (proposed→    (applied → current/
             spec)                                     accepted,     либо epic)
                                                        drafted)
  1. Лид оставляет бриф. Клиент пишет 50–2000 символов на /start-project. Бриф попадает в inbox.
  2. Платформ-овнер принимает лида. Создаётся project + его spec. В спек заводится первая идея — бриф клиента, status=captured, source=brief.
  3. Открывается чат. Лид заходит по magic-link, видит таймлайн спека с одной идеей. Срабатывает первый ход Muse Architect. Идея переходит в discussing.
  4. Идея обрабатывается в диалоге. Человек и Muse обсуждают идею в её чате.
  5. Muse генерирует файлы. Когда идея созрела, Muse генерирует proposal/design/tasks и предлагает их добавить. Файлы: proposed → человек принимает → accepted. Идея переходит в drafted — это уже формальный change.
  6. Файлы обсуждаются и правятся. Диалог продолжается, файлы дорабатываются. Идея остаётся в drafted.
  7. Транзакция применения. Когда change готов:
    • либо файлы вписываются в спек (openspec/current/), идея → applied, папка change → archive/;
    • либо, если идея крупная, она резолвится как epic и порождает дочерние идеи.
  8. Спек подрос. На тот же таймлайн добавляются новые идеи — флоу повторяется. Так спек эволюционирует от первой идеи до зрелого документа.

13Связь с OpenSpec и Git

Как стадии идеи соответствуют трём папкам OpenSpec.

OpenSpec — это методология и git-дерево openspec/. Идея в БД и папка в git — это два дома одной сущности на разных стадиях зрелости.

Папка OpenSpecСтадия идеиСмысл
—captured / discussing Идея существует только как строка в БД; файлов ещё нет либо они в черновике.
openspec/changes/<slug>/drafted Идея материализовалась в формальный change — папка с triplet-ом.
openspec/current/applied Change влит — записан в живой спек.
openspec/archive/applied Применённый change уходит в архив как историческая запись.

Это и есть «транзакция, которая добавляет файлы в current» из обсуждения — переход drafted → applied.

14Стоимость и лимиты

Как учитываются деньги, потраченные на идею.

Каждая идея несёт накопитель стоимости. Тарифицируется и токен-стоимость ходов Muse, и стоимость вызовов инструментов.

МеханизмОписание
cost_centsНакопленная стоимость идеи: ходы + инструменты, одной колонкой.
cost_cap_centsКэп на идею. Текущее значение — $50 (5000¢).
дневной учётОтдельная агрегатная поверхность дневной стоимости (сегодня canvas_daily_costs).
продление кэпаКэп можно расширить явным действием — фиксируется отметкой времени.

Механизм существует сегодня и переносится в новую модель как есть — накопитель и кэп просто переезжают на сущность idea.

16План миграции и риски

Как это безопасно выкатить.

16.1 Масштаб

~110+ файлов кода в apps/ ссылаются на canvas_*, плюс capability-спек openspec/current/canvas.md, плюс RLS-политики, индексы, триггеры, 5 функций и 2 енума.

16.2 Механика

Это атомарная миграция: ALTER TABLE … RENAME по таблицам и колонкам. Важная деталь — имена констрейнтов, индексов, RLS-политик и триггеров не переименовываются автоматически (это строки), их нужно переименовать явно, иначе схема останется с устаревшими именами.

16.3 Риски

РискМитигация
RENAME берёт блокировку ACCESS EXCLUSIVE. Блокировка короткая; выполняется в окне низкой нагрузки.
Старый код мгновенно ломается о новые имена. Деплой кода и миграции — одной транзакцией; rolling-деплой не применим.
Конфликт с in-flight изменениями. Ре-архитектура идёт до старта реализации эпика Ideas — см. раздел 18.
Дрейф schema_migrations. После применения миграции сверить version с именем файла.

18Влияние на эпик Ideas

9 заскаффолженных changes нужно переоснова́ть.

В openspec/changes/ сейчас лежат 9 заскаффолженных changes эпика Ideas (add-ideas-rail-view + 8 зависимых). Все они построены на старом зерне: «идея = стикер, много на канвас, rail внутри канваса». Целевая модель («идея = корневая единица изменения») этому зерну противоречит.

Код эпика ещё не написан — только спек-скаффолд. Поэтому эпик Ideas нужно переоснова́ть на новую модель, а не выбрасывать: жизненный цикл (status), провенанс (source), коммиты, привязка к спек-путям, agent-proposals — всё это переносится на корневую ideas, а не на доску стикеров.

19Следующие шаги

Как этот документ становится реализацией.

  1. Закрыть открытые решения раздела 17 (минимум вопросы 1 и 2).
  2. Завести Linear-issue на ре-архитектуру.
  3. muse draft → формальный OpenSpec-change: proposal.md + design.md + tasks.md. Этот документ — источник для design.md.
  4. Change поглощает: переименование canvas → idea/spec, новую таблицу specs, удаление доски стикеров, переоснову эпика Ideas.
