# Архитектура спека-документа

> Как устроен чат, как живут идеи и что происходит со спеком по дороге — от первой
> идеи на онбординге до момента, когда изменение записано в `openspec/current/`.
>
> Документ описывает **текущее состояние** (как работает сейчас) и **целевую
> архитектуру** (как должно работать). Это аналитический артефакт — основа для
> будущего OpenSpec-change на ре-архитектуру; формальный `design.md` будет
> дистиллирован из него.

**Легенда статусов решений:**

- ✅ — подтверждено в обсуждении
- 🔷 — рекомендация (принимается по умолчанию, можно оспорить)
- ❓ — открытое решение, требует ответа до начала реализации

## 1. Зачем этот документ

Сегодня ядро воркспейса студии — это «канвас»: таблица `canvas_sessions` и
маршрут `/spec/[id]`. Название «canvas» исторично и не отражает того, чем эта
сущность является на самом деле. За разговором выяснилось, что текущая модель
страдает тремя проблемами:

1. **Неверные имена.** «Canvas» — это не холст. «Session» — не сессия (сущность
   живёт неделями, копит стоимость, мержится PR-ом). `mode='project'` — не
   проект.
2. **Размытая иерархия.** Нет уровня «спек» как такового; «канвас» одновременно
   притворяется и одним изменением, и проектным воркспейсом.
3. **Слово «идея» перегружено.** В схеме `canvas_ideas` — это стикеры на доске;
   в голове у нас идея — это единица изменения. Это два разных понятия под одним
   словом.

Цель документа — зафиксировать **правильную** модель, на которой строится вся
дальнейшая работа.

## 2. Словарь

Термины фиксируются здесь и дальше не плывут.

| Термин | OpenSpec-эквивалент | Что это |
|---|---|---|
| **project** | репозиторий / engagement | Контейнер: клиент, репозиторий, Linear-команда, участники. Одна студия, один клиент = один проект. |
| **spec** | `openspec/current/` | Живая спецификация проекта — «что истинно сейчас». Источник истины. Основное дерево, у которого есть идеи. |
| **idea** | change (`openspec/changes/<slug>/`) | Предложенное изменение спека. Одна идея = одно изменение = одна ветка = один PR. |
| **change** | — | Не отдельная сущность. Это **поздняя стадия идеи** — когда у неё уже есть файлы `proposal/design/tasks`. |
| **epic** | epic-change | Не отдельная сущность. Это **идея с детьми** — большая идея, которая вместо прямого применения порождает дочерние идеи. |
| **message** | — | Сообщение в диалоге по конкретной идее. |
| **file** | артефакт изменения | Файл, который Muse сгенерировал для идеи: `proposal.md`, `design.md`, `tasks.md` и прочие. |
| **timeline** | — | Лента идей спека по времени. Это и есть «чат» на уровне спека. |

**Ключевая идея словаря:** `idea`, `change` и `epic` — это **одна сущность**
(`ideas`) на разных стадиях и в разных конфигурациях. «Change» — стадия зрелости.
«Epic» — идея, у которой есть дочерние идеи. Никаких параллельных таблиц.

## 3. Модель данных (целевая)

```
project ──┐  engagement: клиент, репозиторий, Linear-команда, участники
          │
          └─ spec          живая спецификация — «что истинно сейчас»
                │           (= openspec/current/)
                │
                ├─ idea ────┐  предложенное изменение спека
                │           │
                │           ├─ messages   диалог по этой идее
                │           └─ files      proposal / design / tasks этой идеи
                │
                └─ idea (parent_id → idea)   epic = идея с дочерними идеями
```

**Кардинальности:**

- `project` 1 — 1 `spec` — у проекта одна спецификация
- `spec` 1 — N `idea` — много предложенных изменений против неё
- `idea` 1 — N `messages` — у идеи свой диалог
- `idea` 1 — N `files` — у идеи свои артефакты
- `idea` 0 — 1 `parent idea` — само-ссылка; заполнена у дочерних идей эпика

✅ Дерево `project → spec → idea → {messages, files}` подтверждено.
✅ `messages` и `files` цепляются к идее (FK `idea_id`), не к спеку.

## 4. Как это устроено сегодня

Сегодня в БД **12 таблиц** с префиксом `canvas_`:

```
canvas_sessions              ← корневая: один «канвас»
canvas_messages              ← диалог
canvas_files                 ← файлы (proposal/design/tasks и др.)
canvas_ideas                 ← доска стикеров (x/y-сетка, без жизненного цикла)
canvas_invitees              ← доступ
canvas_excluded_members      ← доступ
canvas_project_instructions  ← постоянный контекст для ходов Muse
canvas_tool_uses             ← аудит вызовов инструментов
canvas_proposals             ← Telegram-поверхность
canvas_attachments           ← вложения
canvas_daily_costs           ← дневной учёт стоимости
canvas_stage_events          ← события стадий
```

**`canvas_sessions` сегодня — это уже одно изменение.** Доказательство:
конвертация `convertToProjectAction` создаёт **одну** git-ветку `canvas/<slug>`,
коммитит файлы как дерево, открывает **один** PR. Один канвас = одна ветка = один
PR = одна папка `openspec/changes/<slug>/`. То есть «канвас» по факту и есть
«идея», просто названная неправильно.

**Три неверных имени:**

| Сегодня | Проблема | Должно быть |
|---|---|---|
| `canvas_*` | «холст» — ничего не описывает | `idea` / `spec` |
| `..._sessions` | не сессия — долгоживущий документ | `ideas` |
| `mode='project'` | перегружает слово (есть таблица `projects`) | `mode` = `draft \| branched \| ...` |

**`canvas_ideas` сегодня** — это спатиальная доска стикеров: `content`, `x`/`y`,
`color`, `cluster_id`. Никакого жизненного цикла. Это **не** та «идея», которой
оперирует целевая модель. В целевой модели доска стикеров уходит (см. § 9).

**Онбординг сегодня** (зашипилось 2026-05-21): лид пишет бриф на `/start-project`
→ строка в `inbox` → платформ-овнер принимает (`acceptLeadAction`) → создаётся
`canvas_sessions` + первое `canvas_messages` с `kind='intake_brief'` → при первом
открытии срабатывает `runOnboardingTurn`, и Muse Architect отвечает на языке лида.
То есть бриф сегодня — это **просто первое сообщение чата**, не структурированная
идея.

## 5. Жизненный цикл идеи

Идея — это **одна строка** в `ideas` с полем `status`. Стадии ложатся ровно на
три папки OpenSpec.

```
captured  ──▶  discussing  ──▶  drafted  ──▶  applied   ─▶  current/  (в спек)
                                       └────▶  epic     ─▶  дочерние идеи
                          └──▶  rejected
```

| Стадия | Что происходит | Где живёт |
|---|---|---|
| `captured` | Идея появилась на таймлайне спека, ещё не обсуждалась. Сюда попадает бриф с онбординга. | строка `ideas` |
| `discussing` | Идёт диалог с Muse в чате. | `ideas` + `messages` |
| `drafted` | У идеи есть файлы `proposal/design/tasks` — это уже формальный **change**, материализованный в `openspec/changes/<slug>/`. | `ideas` + `files` + папка `changes/` |
| `applied` | Транзакция влила файлы в спек (`openspec/current/`), идея ушла в `archive/`. | `current/` + `archive/` |
| `epic` | Идея слишком крупная — вместо применения порождает дочерние идеи. | `ideas` + дочерние строки |
| `rejected` | Идея не пошла. | строка `ideas` |

**Перевод стадий на язык разговора:**

- «первая идея с онбординга» → `captured`
- «обрабатывается в чате» → `discussing`
- «у идеи появляются файлы, Muse предлагает» → `discussing → drafted`
- «транзакция добавляет файлы в `current`» → `drafted → applied`
- «или в epic» → `drafted → epic`

🔷 Названия стадий — рекомендация; можно подобрать другие.

## 6. Жизненный цикл файла внутри идеи

У файла свой мини-цикл — Muse **не пишет файл молча**, он предлагает.

```
proposed  ──▶  accepted  ──▶  [обсуждается, правится]  ──▶  applied
(Muse          (человек                                    (транзакция
 сгенерил,      принял,                                      вписала
 предлагает     файл в черновике                             в current/)
 добавить)      идеи)
```

| Стадия файла | Что это |
|---|---|
| `proposed` | Muse сгенерировал файл и предлагает добавить его к идее. Человек видит карточку Approve / Reject. |
| `accepted` | Человек принял — файл стал частью черновика идеи (сегодняшний аналог: `canvas_files.is_pending=true`). |
| `applied` | Транзакция применения идеи вписала файл в спек. |
| `rejected` | Предложение отклонено. |

Паттерн «агент предлагает — человек подтверждает» уже спроектирован в эпике Ideas
(`agent_proposals`, карточка `IdeaProposalCard` с Approve/Reject) и переезжает в
эту модель без изменений.

«Диалог по файлам» = сообщения идеи могут ссылаться на конкретный файл (поле
`file_ref` у сообщения). Отдельная таблица «сообщения файла» не нужна — диалог
один, у идеи, просто с указателями на файлы.

🔷 Стадии файла — рекомендация.

## 7. Чат как таймлайн спека

Чат — это **таймлайн спека**: лента идей по времени.

```
СПЕК (чат-поверхность)
│
├─ idea #1   бриф с онбординга          captured → discussing → drafted → applied
├─ idea #2   «добавить страницу X»      discussing
├─ idea #3   «переписать модель Y»      epic ──┬─ idea #3.1
│                                              └─ idea #3.2
└─ idea #4   ...                        captured
```

- На уровне **спека** «чат» — это упорядоченный по времени список идей
  (`ideas ORDER BY created_at`). Первая идея — бриф с онбординга.
- Открываешь идею → внутри её собственный диалог (`messages`) и её файлы
  (`files`).
- Новые идеи добавляются на ту же ленту, обсуждаются, дозревают до changes.

✅ Сообщения принадлежат идее; спек-таймлайн = идеи спека по порядку.
🔷 Альтернатива (один сплошной поток сообщений на спек, идеи как «якоря» на нём)
отклонена — она размывает идею по диапазону сообщений.

## 8. Сквозной флоу: от онбординга до спека

Конкретный сценарий — что происходит со спеком по дороге.

1. **Лид оставляет бриф.** Клиент пишет 50–2000 символов на `/start-project`.
   Бриф попадает в `inbox`.

2. **Платформ-овнер принимает лида.** Создаётся `project` + его `spec`.
   В спек заводится **первая идея** — бриф клиента, `status=captured`,
   `source=brief`. Это и есть «первая идея, которая заходит в спек на онбординге».

3. **Открывается чат.** Лид заходит по magic-link и видит таймлайн спека с одной
   идеей. Срабатывает первый ход Muse Architect — он переформулирует бриф и
   задаёт уточняющий вопрос (на языке лида). Идея переходит в `discussing`.

4. **Идея обрабатывается в диалоге.** Человек и Muse обсуждают идею в её чате.

5. **Muse генерирует файлы.** Когда идея созрела, Muse генерирует
   `proposal.md` / `design.md` / `tasks.md` и **предлагает** их добавить. Каждый
   файл — `proposed`. Человек принимает → `accepted`. Идея переходит в `drafted`:
   теперь это формальный **change**.

6. **Файлы обсуждаются и правятся.** Диалог продолжается, файлы дорабатываются.
   Идея остаётся в `drafted`.

7. **Транзакция применения.** Когда change готов, транзакция:
   - либо вписывает файлы в спек (`openspec/current/`), идея → `applied`,
     папка change → `openspec/archive/`;
   - либо, если идея крупная, резолвит её как `epic` — идея порождает дочерние
     идеи, у каждой свой полный цикл.

8. **Спек подрос.** Дальше на тот же таймлайн добавляются новые идеи — флоу
   повторяется. Так спек эволюционирует от первой идеи до зрелого документа.

## 9. Что меняется в схеме

Карта переименования (`canvas → idea/spec`):

| Сегодня | Цель | Примечание |
|---|---|---|
| `canvas_sessions` | `ideas` | Корневая единица изменения. |
| `canvas_messages` | `messages` | FK перецепляется `canvas_id → idea_id`. |
| `canvas_files` | `files` | FK перецепляется `canvas_id → idea_id`; добавляется `status` (proposed/accepted/applied). |
| `canvas_ideas` | **удаляется** | Доска стикеров уходит; свободные мысли становятся либо сообщениями, либо под-идеями. |
| колонка `canvas_id` | `idea_id` | ~323 вхождения по всей схеме. |
| — | `specs` | **Новая** тонкая таблица-якорь, одна строка на проект. |
| `ideas.parent_id` | — | **Новая** само-ссылка для эпиков. |
| `canvas_invitees`, `canvas_excluded_members` | → уровень `spec` | Доступ логичнее привязать к спеку, не к отдельной идее. ❓ под-дизайн. |
| `canvas_proposals`, `canvas_tool_uses`, `canvas_daily_costs`, `canvas_stage_events`, `canvas_attachments`, `canvas_project_instructions` | переименовать, перецепить | Остаются дочерними; точная привязка (idea vs spec) — под-дизайн. |
| функции `has_canvas_role` и др. (5 шт.) | `has_idea_role` / `has_spec_role` | По смыслу. |
| енумы `canvas_role`, `canvas_invitee_role` | `spec_role` / `idea_role` | По смыслу. |

Масштаб: ~110+ файлов кода в `apps/` ссылаются на `canvas_*`, плюс capability-спек
`openspec/current/canvas.md`, плюс RLS-политики, индексы, триггеры. Это
**атомарная** миграция: переименование таблиц + колонок + констрейнтов + индексов
+ политик, регенерация `packages/db/types.ts`, обновление кода и спеков. Деплой
кода и миграции — одной транзакцией (старый код мгновенно ломается о новые имена).

## 11. Влияние на эпик Ideas

В `openspec/changes/` сейчас лежат **9 заскаффолженных changes** эпика Ideas
(`add-ideas-rail-view` + 8 зависимых). Все они построены на **старом** зерне:
«идея = стикер, много на канвас, rail внутри канваса». Целевая модель
(«идея = корневая единица изменения») этому зерну противоречит.

Хорошая новость: код эпика ещё не написан — только спек-скаффолд. Поэтому эпик
Ideas нужно **переоснова́ть** на новую модель, а не выбрасывать: лайфсайкл
(`status`), провенанс (`source`), коммиты, привязка к спек-путям, agent-proposals
— всё это переносится на корневую `ideas`, а не на доску стикеров.
