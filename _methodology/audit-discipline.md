# Audit discipline — verification gate for live-fetch audits

> When an audit is produced by fetching a deployed surface rather than reading
> the repository, its findings are hypotheses — not facts. This document is the
> gate that turns hypotheses into actionable findings.

---

## Why this exists

On 2026-05-18 a five-specialist marketing audit of `rapoport.studio` was
generated by `web_fetch` of the live site. It produced ~22 findings across
code, brand, conversion, SEO and IA. A verification pass against repo code
refuted most of the P0/P1 claims — the fetch had hit a stale or preview
deploy, and `web_fetch` strips `<script>` tags, so every JSON-LD / metadata
claim was a guess. Six findings survived; the rest were noise.

Acting on the raw audit — filing ~22 Linear issues, opening PRs — would have
spent real effort on non-bugs and eroded trust in the audit channel. The
verification pass cost a fraction of that. Worked example:
`openspec/_methodology/research/home-five-specialist-audit.md`.

## The rule

**Any audit that asserts a structural or semantic finding and was NOT produced
by reading the repository must pass a repo-verification gate before its
findings are filed as Linear issues or acted on in a PR.**

A *structural or semantic finding* is any claim about: i18n / locale parity,
metadata or `<head>` content, JSON-LD / structured data, routing or link
targets, component behaviour, DOM structure, or file / spec contents.

*Not produced by reading the repository* covers: `web_fetch` of a deployed
site, screenshots, black-box QA, third-party audit tools, and any audit run
against an environment whose code the auditor did not read.

## What "verified" means

For each asserted finding:

- **Confirmed** — cite the exact `file:line` that proves the claim.
- **Refuted** — name the `file:line` that disproves the claim. Never a bare
  "checked, it was fine".
- **Unverifiable** — needs a browser, a database, or runtime state; label it as
  such and exclude it from the actionable set.

Only the confirmed set may become Linear issues or PR scope. The reviewed
artifact is the *verified* audit document (confirmed / refuted / unverifiable),
not the raw fetch output.

## Scope

- **Gated:** live-fetch audits, black-box audits, tool-generated audits.
- **Exempt:** audits performed directly against the repository — reading source
  at `file:line` *is* the verification — and live-fetch audits that make only
  subjective findings (visual taste, copy tone) with no structural claim.

A stale-source risk that cannot be diagnosed (e.g. why a live fetch returned
old HTML) is recorded as an explicit open risk in the verified audit, not
silently dropped.
